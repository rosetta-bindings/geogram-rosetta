# Geogram x Rosetta

Language bindings for [Geogram](https://github.com/BrunoLevy/geogram) (a programming library with geometric algorithms) generated by [rosetta](../rosetta) — **Python**, **Node.js**, **WebAssembly** and **TypeScript** from a single C++ facade and one `manifest.json`.

![Geogram](media/screen.jpg)

## Design

`GEO::Mesh` cannot be bound directly by rosetta: its whole data API lives in public member objects (`mesh.vertices`, `mesh.facets`, ...) that hold a `Mesh&` back-reference (non copy-assignable) and expose raw `double*` / `index_t*` accessors and `GEO::vector` containers — none of which a scripting backend can marshal.

This wrapper exposes the same geometry as flat `std::vector` arrays and hides `GEO::Mesh` behind a pImpl so the reflection generator never has to parse **geogram** headers.

Vertices are flat `[x0,y0,z0, x1,y1,z1, ...]` arrays; triangles are flat arrays of 3 vertex indices per triangle. Polygonal facets are fan-triangulated on output by `triangles()` / `tex_coords()`.

Two more reasons the manifest alone is not enough (compare with [pmp-rosetta](../pmp-rosetta), which binds `pmp::SurfaceMesh` directly): geogram's algorithms expect `GEO::initialize()` + the `CmdLine` argument groups they read their tuning parameters from, and several key entry points don't reflect well — `mesh_load`/`mesh_save` and the boolean helpers are overloaded (`^^name` is ill-formed for overloads), `CSGCompiler::compile_*` returns `std::shared_ptr<Mesh>`, and the atlas UVs land in a facet-corner *attribute*.

So the facade is deliberately **marshalling-only** (~250 lines, no algorithmic code): [`src/Mesh.{h,cpp}`](src/Mesh.h) wraps `GEO::Mesh`, and [`src/algorithms.{h,cpp}`](src/algorithms.h) exposes the geogram algorithms as one-line delegating free functions that mirror geogram's names.

## What is bound

| Family | Functions |
|---|---|
| Surface reconstruction | `co3ne_smooth`, `co3ne_smooth_and_reconstruct` (Co3Ne: smoothing + co-cone triangulation of a point cloud) |
| Remeshing | `remesh_smooth` (isotropic CVT remeshing, Lloyd + Newton), `mesh_repair`, `fill_holes`, `tessellate_facets` |
| Parameterization & texturing | `make_atlas` (charts + LSCM/ABF/spectral/projection + tetris/xatlas packing), `get_charts`, `Mesh.tex_coords()` |
| Intersections & Booleans | `mesh_union`, `mesh_intersection`, `mesh_difference`, `mesh_remove_intersections`, `mesh_facets_have_intersection` (exact arithmetic) |
| CSG | `csg_evaluate_string`, `csg_evaluate_file` (OpenSCAD `.csg` programs: `sphere`, `cube`, `cylinder`, `union`, `difference`, `intersection`, `multmatrix`, `hull`, `minkowski`, `linear_extrude`, ...) |

## Layout

```
manifest.json           what to bind (facade class + functions, geogram sources, targets)
src/Mesh.{h,cpp}        marshalling wrapper around GEO::Mesh
src/algorithms.{h,cpp}  one-line delegations to the GEO:: algorithms
src/geogram/version.h   stand-in for geogram's CMake-generated version header
gen/                    (generated) the reflection-based generator project
generator               (built) the binding generator executable
bindings/               (generated) python / node / wasm / typescript targets
example_python.py       end-to-end demo (CSG → booleans → remesh → atlas → reconstruct)
example_node.js         same demo through the N-API addon
example_wasm.js         same demo through embind/WASM
index.html              browser demo: 3D viewer (three.js) + geogram algorithms in wasm
data/                   Intergalactic_Spaceship.obj — sample model for the browser demo
```

The geogram sources are compiled **straight into every binding target** (manifest `user_sources` globs — no prebuilt `libgeogram`: the wasm target needs the sources anyway, and a native lib built by another toolchain would risk an ABI mismatch). Two manifest fields configure the build: `compile_definitions` (`GEOGRAM_USE_BUILTIN_DEPS`, `GEOGRAM_WITH_HLBFGS`) and, on the wasm target, `link_options: ["-lnodefs.js"]` (geogram mounts NODEFS from `GEO::initialize()` under Node).

## Prerequisites

- clang-p2996 (C++26 reflection fork) at `~/devs/c++/clang-p2996/build` — used **only** to build the generator; the bindings themselves compile with stock toolchains (`*-expanded` targets).
- `../rosetta` checked out as a sibling.
- geogram under `extern/geogram` **with submodules** (the bootstrap CMakeLists fetches it, or: `git clone --recurse-submodules --depth 1 https://github.com/BrunoLevy/geogram extern/geogram`).
- Python 3 + `pybind11` (`pip install pybind11`) for the Python target.
- Node.js (`cmake-js` is installed by `npm i`) for the Node target.
- emsdk (`source ~/emsdk/emsdk_env.sh`) for the WASM target.

## Build

```bash
# 0. One-time bootstrap: fetch geogram, build rosetta_gen
cmake -B build && cmake --build build

# 1. Generate the generator project from manifest.json, then build it
../rosetta/bin/rosetta_gen manifest.json gen
cmake -S gen -B gen/build && cmake --build gen/build

# 2. Generate all binding projects (bindings/…)
./generator bindings

# 3. Python
cmake -S bindings/python-expanded -B bindings/python-expanded/build
cmake --build bindings/python-expanded/build -j
python3 example_python.py

# 4. Node.js
(cd bindings/node-expanded && npm i && npm run build)
node example_node.js

# 5. WebAssembly
source ~/emsdk/emsdk_env.sh
emcmake cmake -S bindings/wasm-expanded -B bindings/wasm-expanded/build
cmake --build bindings/wasm-expanded/build -j
node example_wasm.js

# 6. TypeScript declarations (no build needed)
cat bindings/typescript/geogram.d.ts
```

All three examples run the same pipeline — CSG (sphere minus cylinder), the three boolean operations on two overlapping spheres, CVT remeshing of the union to 5000 vertices, LSCM/xatlas UV-atlas generation with UV read-back, then Co3Ne reconstruction of the surface from its bare point cloud — and print the same counts:

```
CSG        : sphere minus cylinder -> 470 vertices, 940 facets
booleans   : union 1580, intersection 812, difference 1196 facets
remeshing  : union remeshed to 5000 vertices, 9996 facets
texturing  : 7 charts, 29988 UV corners in [0.000, 1.000]
reconstruct: Co3Ne rebuilt 9996 facets from the point cloud
```

## Browser demo (`index.html`)

A three.js viewer that runs the wasm binding on a real model in the browser
(screenshot above). Build the WASM target (step 5), then serve the project
root over HTTP and open the page:

```bash
python3 -m http.server        # then http://localhost:8000/
```

- **Load** fetches `data/Intergalactic_Spaceship.obj` (27.5k vertices /
  55k triangles); the file picker accepts any local `.obj`. The OBJ is
  parsed in JS (fan-triangulated) and pushed into a `georo::Mesh` through
  `set_surface` — no filesystem involved.
- **Repair** (`mesh_repair`), **Fill holes** (`fill_holes`) and
  **Remesh (CVT)** (`remesh_smooth`, with points / Lloyd / Newton controls)
  modify the wasm-side mesh; **Reset** restores the pristine scan
  (`Mesh.assign`).
- Display toggles: plain shaded surface, **wireframe** overlay, vertex
  **points**, flat shading; orbit / zoom / pan with the mouse.

Everything runs single-threaded in wasm on the main thread — on the bundled
spaceship, repair takes ~0.3 s and the default remesh (10k points,
5 Lloyd + 10 Newton) ~20 s; a 1.1M-triangle scan remeshes in ≈2 min
(Newton = 0 is about 3x faster). The log panel prints per-operation timings.

## Notes

- **CSG dialect**: geogram evaluates the *compiled* OpenSCAD format (`openscad model.scad -o model.csg`). High-level transforms are OpenSCAD sugar — write `multmatrix([[1,0,0,tx],[0,1,0,ty],[0,0,1,tz],[0,0,0,1]]) { ... }` instead of `translate([tx,ty,tz])`.
- `initialize(verbose)` runs lazily before any algorithm; call `initialize(true)` first to see geogram's logger (OpenNL solver output is routed through the logger too, so it honors the quiet flag).
- In the WASM build, embind does not auto-convert JS arrays: use `Module.vector_double` / `Module.vector_int` and call `.delete()` on vectors and meshes (see `example_wasm.js`).
- `mesh_union` / `mesh_intersection` / `mesh_difference` require closed surfaces without self-intersections (that is what the CSG primitives produce; use `mesh_repair` / `mesh_remove_intersections` on wild input).
- The Node target needs the `from_napi` by-reference fix for wrapped classes in rosetta's `node_runtime.h` (bound functions mutate `Mesh&` out-parameters; the old by-value unwrap didn't compile and would have shallow-copied the pImpl).
